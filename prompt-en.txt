You are an AI engineering expert. Build a complete AI collaboration documentation system for the current project.

## Goal

Establish a documentation structure in the project that enables AI Agents (cross-editor: Copilot, Cursor, Trae, etc.) to collaborate efficiently. This includes: project guide, coding standards, reusable skills, custom Agents, Prompt templates, and development progress tracking.

## Requirements

### Step 1: Analyze the Project

#### 1.1 Scan Boundaries

Define the analysis scope first:
- **Ignore directories**: `node_modules`, `dist`, `build`, `.next`, `coverage`, `vendor`, `generated`, `__pycache__`, `.cache`, lock files, and other build artifacts / dependencies / caches / generated files
- **Priority reading**: Root README, package manifests (package.json / go.mod / Cargo.toml / pom.xml, etc.), entry files, core business directories, test directories, CI configuration
- **Large project strategy**: If the project has more than 20 core modules or a massive codebase, sample core modules first — do not attempt a full scan. Place remaining modules in a backlog for later coverage
- **Core module identification**: Prioritize based on entry point reference chains, main business directories, build configuration, CI coverage scope, test coverage density, and modules explicitly listed in README
- **"Massive codebase" definition**: Refers to projects with obvious multi-subsystem characteristics, complex directory hierarchies, scattered configs and entry points, or where a single analysis pass cannot form a reliable global understanding

#### 1.2 Analysis Content

Within the scan boundaries, understand:
- Project purpose and core responsibilities
- Tech stack (languages, frameworks, build tools)
- Module breakdown and their responsibilities
- Existing coding conventions and architectural decisions
- Current project phase (initial build / active development / maintenance)
- Recurring development operation patterns

#### 1.3 Information Source Priority

When different sources conflict, judge by this priority:

**Build config / build scripts / CI config / current source structure > Tests > README > other historical docs > comments > issues / commits**

#### 1.4 Degradation Strategy

When encountering uncertainty during analysis:
- If project standards are inconsistent, document the "current state" rather than forcing unification
- If architectural decisions cannot be confirmed, mark as `<!-- TBD -->`
- If the project is too large for full analysis, cover core modules only; list the rest in a backlog
- If existing standards conflict, explain the source of conflict and each side's current state — do not arbitrate

### Step 2: Output Analysis Report

**Do not create files immediately.** First output the following for confirmation:
1. Project understanding summary (purpose, tech stack, modules, phase, core conventions)
2. Proposed `.ai/` directory structure and file list
3. Expected content summary for each file (one sentence)
4. Identified uncertainties and items requiring confirmation
5. For large projects, state the current round's coverage scope and backlog

Each key judgment must include at least one evidence source (e.g., which directory, config file, test, or entry file it came from) — detailed quotes are not required, but it must be traceable.

Proceed to Step 3 after user confirmation. If the user explicitly says "just create them", or the project meets most of the following conditions, skip confirmation and execute directly:
- Core modules ≤ 10
- Tech stack involves no more than 2–3 categories
- Root directory structure is clear, no obvious multi-subsystem / monorepo characteristics
- Existing documentation largely matches the code

### Step 3: Create Files by Priority

#### 3.1 Output Priority

Create in the following order, ensuring core file quality first:

**First priority (required):**
1. `AGENTS.md` — Project overview and navigation entry (auto-loaded, doubles as guide)
2. `.ai/instructions/ai-workflow.md` — Documentation update standards
3. Core instructions (2–4 high-value standards files)
4. `MEMORY.md` — Development progress tracking

**Second priority (as needed):**
5. `.ai/docs/` — Feature design documents
6. `.ai/skills/` — High-frequency skills (2–3)
7. `.ai/agents/` — At least one read-only review Agent
8. `.ai/prompts/` — Common Prompt templates

**Third priority (editor adapters):**
9. `.github/copilot-instructions.md` — Copilot repo-level entry
10. `.github/instructions/*.instructions.md` — Copilot path-level rules
11. `.cursor/rules/*.mdc` — Cursor rule entries

#### 3.2 Safe Write Rules

- If a target file already exists, prefer to **merge and supplement** rather than blindly overwrite. When merging, preserve the original file structure and fill in missing sections; if original content conflicts with current analysis, keep the original text and mark with `<!-- TBD -->`
- Write uncertain content as `<!-- TBD -->` — do not disguise it as confirmed conclusions
- Specific skill / agent / prompt / doc files are created as needed, but **all directories must be created**
- Each on-demand directory (docs / skills / agents / prompts) must contain a `README.md` explaining the directory's purpose, file naming conventions, and creation criteria — so users aren't confused by the directory's existence
- Apart from directory READMEs, do not create placeholder business files just to fill the structure; on-demand files should only be created when supported by project evidence
- If the project already has similar documentation, new documents should **link to existing docs** rather than duplicate descriptions

#### 3.3 File Structure

All directories must be created. Files marked `[as needed]` are created based on actual project requirements.

```
AGENTS.md                     — Project overview and navigation entry (root, ≤300 lines, auto-loaded)
MEMORY.md                     — Development progress tracking (root directory)
.ai/
  instructions/               — Domain-specific coding standards
    ai-workflow.md            — Documentation update standards (required)
    *.md                      — Other standards files (as needed)
  docs/                       — Feature design and tracking documents
    README.md                 — Directory guide: purpose, file naming conventions, creation criteria
    changelog.md              — MEMORY.md history archive index (as needed)
    archive/                  — Archive for invalidated prompts / skills / docs (as needed)
    *.md                      — Design documents (as needed)
  skills/                     — Reusable operation templates
    README.md                 — Directory guide: what skills are, extraction criteria, structure requirements
    <skill-name>/SKILL.md     — Specific skills (as needed)
  agents/                     — Custom Agents
    README.md                 — Directory guide: Agent definition format, tool restriction guidelines
    <name>.agent.md           — Specific Agents (as needed)
  prompts/                    — Prompt shortcut templates
    README.md                 — Directory guide: Prompt format requirements, referencing conventions
    <name>.prompt.md          — Specific Prompts (as needed)
```

> Deep-review workflow artifacts (`tests/`, `deep-review.agent.md`, `deep-review.prompt.md`, and the `## Review Log` section in `MEMORY.md`) are an **opt-in capability initiated by the user**. **Do not create them by default and do not ask about them proactively.** Create them incrementally per the "Deep-Review Workflow Artifacts" section only when the user explicitly requests it (e.g., "enable review workflow", "add deep review", `/review`).

### Step 4: Format Requirements for Each File

#### AGENTS.md (root directory — sole auto-loaded entry point)

AGENTS.md is the file auto-loaded by editors (Copilot / Cursor, etc.) for every conversation, so it serves dual roles: "project overview" and "navigation guide". A separate `.ai-instructions` guide file is no longer created.

- Title: `# Project Guidelines`

**Top half — Project summary:**
- Project positioning (1-2 sentences)
- Tech stack (list)
- Core module overview (table — list core modules only, omit non-core)
- Core conventions (4-5 items)

**Bottom half — Documentation navigation:**
- **Required reading**: List MEMORY.md → core instructions in order of importance (only the top 2–3, with a one-line description each)
- **On-demand resources**: A compact table listing key files under `.ai/` docs / skills / agents / prompts with trigger conditions (YAML frontmatter descriptions assist discovery; only list the most frequently used items here)
- **Mandatory workflow**: What to read before starting → update MEMORY.md after completing
- **Language requirements**

Move detailed content to `.ai/docs/`; AGENTS.md serves only as index and navigation. Keep under 300 lines — it's loaded in full for every conversation.

#### .ai/instructions/*.md (coding standards)
- Each file must have YAML frontmatter:
  ```yaml
  ---
  description: "Use when: trigger keyword description"
  applyTo: "glob pattern"    # Optional, auto-attach by file type
  ---
  ```
- One file per concern (backend/frontend/API/testing — don't mix)
- Content should be concise and actionable; use lists, not long paragraphs
- For multi-language projects, split by language or domain
- Must create an `ai-workflow.md` defining documentation update standards

#### .ai/skills/<name>/SKILL.md (skill templates)
- Must have YAML frontmatter:
  ```yaml
  ---
  name: skill-name           # Must match directory name
  description: "Use when: keyword-rich trigger description"
  argument-hint: "input hint"
  ---
  ```
- Body includes: applicable scenarios, project conventions (with code examples), step-by-step procedures, key constraints
- **Skill extraction threshold**: Only extract as a skill when the operation meets ALL of the following:
  - A repeating pattern that appears 3+ times in the project
  - Has clear inputs and outputs
  - Steps are stable and reproducible
  - Contains project-specific constraints or conventions
  - Can be reused for future similar tasks
- Do not create skills for generic operations with no project-specific characteristics (e.g., "read code", "fix bug")

#### .ai/agents/<name>.agent.md (custom Agents)
- Must have YAML frontmatter:
  ```yaml
  ---
  description: "Use when: trigger description"
  name: "agent-name"
  tools: [read, search]      # Minimal tool set
  ---
  ```
- Body defines: role positioning, scope of responsibilities, constraints (what NOT to do), specific checklists or work steps
- Create at least one read-only review Agent (tools: [read, search])
- `tools` is a document-level capability constraint declaration expressing the minimal required capabilities — it does not depend on any specific editor's implementation

#### .ai/prompts/<name>.prompt.md (Prompt templates)
- Must have YAML frontmatter:
  ```yaml
  ---
  description: "One-line purpose description"
  agent: "agent"
  tools: [read, edit, search]
  ---
  ```
- Body uses Markdown links to reference related instructions and SKILL.md files
- Create Prompts for the 3–5 most common operations in the project; if the project has fewer scenarios, fewer than 3 is acceptable — do not create low-value templates just to meet a quota

#### MEMORY.md (development progress)
- Include: current phase, completed items, AI engineering status checklist, follow-up suggestions, **change log**
- Mark last update date in file header
- Keep under 100 lines; archive excess to `.ai/docs/changelog.md`
- **Change log entry format**: one line per entry, short and clear, must include the model name and a timestamp precise to the minute
  ```
  - YYYY-MM-DD HH:MM | <model-name> | <one-sentence description of the change>
  ```
  - Use local timezone, minute precision (do not use date-only)
  - `model-name` identifies the model that made the change, e.g. `claude-opus-4.7`, `gpt-5`, `gemini-2.5-pro`
  - Description ≤ 80 characters; no multi-paragraph text; collapse multi-file changes into a single entry
- When the deep-review workflow is enabled, add a separate `## Review Log` section, one line per round (see deep-review section below)

#### Deep-Review Workflow Artifacts (created only when the user initiates it; not created by default)

##### tests/ directory
- Reuse the existing test directory and framework when present; otherwise create one per language convention
- `tests/README.md` must state: test framework used, unified run command, directory organization, expected coverage
- `ai-review/` (or equivalent subdirectory) holds regression tests added by the review workflow; filenames must be traceable to the triggering review item

##### .ai/agents/deep-review.agent.md
- frontmatter `tools` minimum set: `[read, search, edit, run]` (`run` is required to execute tests)
- Fixed 5-step loop, executed strictly in order:
  1. **Review** — produce an issue list based on `.ai/instructions/` and the latest diff
  2. **Add tests** — write tests under `tests/` for uncovered critical paths
  3. **Run tests** — execute the project test command and collect failures
  4. **Fix failures** — minimal fix targeting failures; **never bypass failures by deleting or skipping tests**
  5. **Review again** — return to step 1, until no new issues and all tests pass
- Exit condition: two consecutive rounds with no new issues and all tests green
- At the end of each round, append one entry to the `## Review Log` section in `MEMORY.md`

##### .ai/prompts/deep-review.prompt.md
- References `deep-review.agent.md` and related instructions
- Provides a trigger entry (e.g., `/review` or a one-line trigger)

##### MEMORY.md Review Log format
```
## Review Log
- YYYY-MM-DD HH:MM | <model-name> | round <N> | <one-sentence summary of changes and conclusion>
```
- Minute precision; `round N` marks the current loop iteration; the conclusion must explicitly state "pass / continue next round"

### Maintenance and Evolution

To ensure the `.ai/` system doesn't become outdated:
- Define in `ai-workflow.md`: which project changes must trigger instruction updates (e.g., tech stack migration, directory restructuring, new modules)
- Archive invalidated prompts and skills to `.ai/docs/archive/` promptly — do not leave them in active directories
- Archive MEMORY.md content exceeding 100 lines to `.ai/docs/changelog.md`
- After any major refactoring, re-evaluate the accuracy of the `.ai/` system

### Large Project Adaptation

If the project is a monorepo or contains multiple subsystems:
- Root-level `.ai/` stores universal standards and navigation
- Each sub-project/sub-module can have its own local `.ai/instructions/`
- Root directory retains only cross-subsystem shared standards and navigation index
- Subsystem-specific standards should not be crammed into root-level instructions
- When root-level standards conflict with subsystem-local standards, the subsystem-local standards take precedence within their scope; root-level standards only provide shared constraints

### Editor Adapter Layer

The `.ai/` system is the source of truth containing all specification content. To improve native rule discovery for Copilot and Cursor, add a thin adapter entry layer.

#### General Principles

- `.ai/` holds all content; adapter layers only write short entries — never a second copy of specifications
- Adapter files' job is to "tell the editor which source files to read" — not to duplicate rules
- Adapter files must always be shorter than their source; keep each under 10–20 lines
- Only modify source files; sync adapter layers only when source files are renamed or paths change

#### Copilot Adapter (required)

Copilot natively recognizes `.github/copilot-instructions.md` (repo-level) and `.github/instructions/*.instructions.md` (path-level).

**`.github/copilot-instructions.md`**:
- Keep to 10–20 lines
- Content: read AGENTS.md first (project overview and navigation guide are already in it), information source priority, TBD marking rule, update MEMORY.md after work, language requirements
- No specific coding standards

**`.github/instructions/*.instructions.md`**:
- Create adapter files only for core `.ai/instructions/*.md` (2–4 files) — do not mirror all
- Filename must end with `.instructions.md`
- Must have YAML frontmatter with `applyTo` matching the source file's `applyTo` coverage
- Body: which source files to read + 2–4 key constraints only

Example:
```md
---
applyTo: "**/*.ts,**/*.tsx,src/frontend/**"
---

Read `AGENTS.md` and `.ai/instructions/frontend.md` before editing these files.

Rules:
- Follow existing component patterns before introducing new abstractions.
- Keep changes minimal and consistent with nearby files.
- Update related tests when behavior changes.
- Mark uncertain decisions as `<!-- TBD -->`.
```

#### Cursor Adapter (recommended)

Cursor recognizes rule files under `.cursor/rules/*.mdc`.

**`.cursor/rules/00-core.mdc`**:
- Set `alwaysApply: true`, `globs: ["**/*"]`
- Content mirrors `copilot-instructions.md`: read AGENTS.md first (navigation guide is already in it), same core constraints

**`.cursor/rules/<domain>.mdc`**:
- Create only for core domains (2–4 files) — do not mirror all
- Set `alwaysApply: false`, use `globs` matching the source file's `applyTo`
- Body: which source files to read + 2–4 key constraints only

Example:
```md
---
description: Frontend file editing rules
globs: ["**/*.tsx", "**/*.jsx", "src/frontend/**"]
alwaysApply: false
---

Before editing these files, read:
- `AGENTS.md`
- `.ai/instructions/frontend.md`

Rules:
- Reuse existing UI patterns before creating new abstractions.
- Keep props and state handling consistent with nearby files.
- Update related tests when UI behavior changes.
```

#### Adapter Mapping

| Source | Copilot Adapter | Cursor Adapter |
|--------|-----------------|----------------|
| `AGENTS.md` (overview + navigation) | Reused directly + referenced in `.github/copilot-instructions.md` | Reused directly + referenced in `00-core.mdc` |
| `.ai/instructions/<domain>.md` | `.github/instructions/<domain>.instructions.md` | `.cursor/rules/<domain>.mdc` |
| `MEMORY.md` | Required update in repo-level entry | Required update in core rule |

#### Sync Discipline

1. **Only modify source**: Specification changes go to `.ai/` first — never directly to adapter layers
2. **Reference sync**: Only sync adapter references when source files are renamed or paths change
3. **No detail in adapters**: Adapters only provide navigation, triggers, and minimal hard rules

### Universal Principles

- **Evidence first**: Documentation content must be based on source code, build config, tests, and CI — never fabricated
- **Concise first**: AGENTS.md and instructions consume context window — shorter is better
- **Keyword-driven discovery**: description is the sole basis for Agent file discovery — keywords must be accurate
- **One file, one concern**: Don't mix test standards with API standards
- **Show don't tell**: Use brief code examples rather than long paragraphs
- **Link don't embed**: Put detailed content in docs; main files only index
- **Cross-editor compatible**: Don't rely on editor-specific mechanisms (like Hooks); file formats must be universal
- **Safe writes**: Prefer supplementing existing files over overwriting; mark uncertainties as TBD; never create empty files
