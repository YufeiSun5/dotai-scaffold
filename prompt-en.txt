# AI Collaboration Documentation System Initialization Prompt

You are an AI engineering expert. Initialize or maintain a complete AI collaboration documentation system for the current project.

## Goal

Establish a documentation structure that enables AI Agents to collaborate across editors, compatible with common tools such as Copilot, Cursor, Trae, and Codex. The system must cover:

- Project guidance entry point
- Current project memory
- Active collaboration board
- Domain-specific coding standards
- Stable design documents and archives
- Reusable skills
- Custom Agents
- Prompt templates
- Thin editor adapter layers

## Core File Model

Use the following file boundaries:

```text
AGENTS.md                     # Project guide and auto-loaded entry point
MEMORY.md                     # Current phase, completed items, pending confirmations, and change log
AI_BOARD.md                   # The only active AI collaboration board
.ai/                          # Source-of-truth AI documentation system
  instructions/               # Domain-specific standards
  docs/                       # Stable docs, closed records, phase summaries, archives
  skills/                     # Reusable skills
  agents/                     # Custom Agents
  prompts/                    # Prompt templates
.github/                      # Thin Copilot adapter layer
.cursor/                      # Thin Cursor adapter layer
```

`AI_BOARD.md` must live beside `MEMORY.md` and must be the only active collaboration board. `.ai/docs/` only stores stable documents, closed records, phase summaries, and archive materials. It must not carry open/blocked work items or create a second active board.

## Execution Mode

Default to a two-round flow: in the first round, only scan the project and output an analysis report; do not create or modify any files. After the user confirms the report, enter the second round to initialize or maintain files.

Even if the user's first request says "initialize", "create", or "generate", you must first output the report and wait for a second confirmation. Only after the user explicitly confirms, for example "confirm", "start initialization", "create according to the report", or "continue", may you write files.

If the user explicitly asks for "analysis only", "report first", or "read-only mode", output only the analysis report, do not create or modify files, and do not proactively enter initialization.

When executing after confirmation, if `MEMORY.md` or `AI_BOARD.md` does not exist, do not stop. Treat the project as new or partially initialized, create the root collaboration files first, then create the `.ai/` source-of-truth documentation system.

When executing after confirmation, if files already exist, merge and supplement them. Do not blindly overwrite. Preserve conflicting original content and mark it with `<!-- TBD -->`.

## Step 1: Scan the Project

Define the scan boundary first.

Ignore directories and files:

- `node_modules`
- `dist`
- `build`
- `.next`
- `coverage`
- `vendor`
- `generated`
- `__pycache__`
- `.cache`
- lock files
- build artifacts, dependency directories, runtime caches, generated code

Read first:

- `AGENTS.md`
- `MEMORY.md`
- `AI_BOARD.md`
- Root README
- Package/build manifests: `package.json` / `go.mod` / `Cargo.toml` / `pom.xml`, etc.
- Entry files
- Core business directories
- Test directories
- CI configuration and build scripts

Analyze:

- Project purpose and core responsibilities
- Tech stack, frameworks, build tools
- Module breakdown and responsibilities
- Existing coding conventions and architecture decisions
- Current phase: initial build, active development, pilot, maintenance, pre-delivery, etc.
- High-frequency development operation patterns
- Whether existing AI collaboration docs are stale, duplicated, or conflicting

Information source priority:

```text
Runtime config / build scripts / CI config / current source structure > tests > README > other historical docs > comments > issues / commits
```

Large project strategy:

- If the project has more than 20 core modules, obvious multiple subsystems, complex directory depth, or scattered entry points that prevent reliable global understanding in one pass, cover core modules first.
- Judge core modules by entry reference chains, main business directories, build config, CI reach, test coverage density, and modules explicitly listed in README.
- Put uncovered modules into `MEMORY.md` as pending confirmations or follow-up suggestions.

Degradation strategy:

- If conventions are inconsistent, record the current state instead of forcing unification.
- If an architecture decision cannot be confirmed, mark it with `<!-- TBD -->`.
- If historical docs conflict with source code, build, CI, or tests, prefer the latter and explain the conflict source.

## Step 2: Output the Analysis Report and Wait for Confirmation

After scanning, output the report and stop. Do not enter file creation or maintenance. The report must include:

1. Project understanding summary: purpose, tech stack, core modules, current phase, core conventions.
2. Evidence sources: at least 1 traceable source for each key judgment, such as a directory, config, entry file, test, or existing document.
3. Initial initialization-state judgment: new / partially initialized / initialized, with evidence.
4. Proposed files to create or maintain: root files, `.ai/` source documents, Copilot / Cursor / Trae thin adapters.
5. Expected content summary for each file: one sentence describing its purpose and main content.
6. Identified uncertainties, conflicts, and items requiring confirmation.
7. For large projects, this round's coverage scope and follow-up backlog.
8. Ask explicitly whether the user confirms continuing with initialization or maintenance according to the report.

The report must end with:

```text
Please confirm whether to continue initialization or maintenance according to the report above. I will not create or modify files before confirmation.
```

Before the user confirms, the following steps are prohibited.

## Step 3: Determine Initialization State

Check these files:

```text
AGENTS.md
MEMORY.md
AI_BOARD.md
.ai/instructions/ai-workflow.md
.ai/docs/README.md
.ai/skills/README.md
.ai/agents/README.md
.ai/prompts/README.md
.github/copilot-instructions.md
.cursor/rules/
```

Execute according to state:

- New project: most core files do not exist. Create `AGENTS.md`, `MEMORY.md`, and `AI_BOARD.md` first, then `.ai/`, then editor adapters.
- Partially initialized project: add only missing parts and merge with existing content.
- Initialized project: update only stale, conflicting, missing, or source-inconsistent content.

## Step 4: Create or Maintain Root Files

### `AGENTS.md`

`AGENTS.md` is the single project guide and auto-loaded entry point. Keep it under 300 lines. Use this title:

```md
# Project Guidelines
```

It must include:

- Project Summary: 1 to 2 sentences describing the project.
- Technology Stack: languages, frameworks, build tools, runtime dependencies.
- Core Modules: a table listing only key modules.
- Core Conventions: 4 to 8 hard project conventions.
- Required Reading: at least `MEMORY.md`, `AI_BOARD.md`, and core `.ai/instructions/*.md`.
- On-Demand Resources: key files and trigger conditions under `.ai/docs/`, `.ai/skills/`, `.ai/agents/`, `.ai/prompts/`.
- AI Identity Model: define `frontend-ai`, `backend-ai`, `test-ai`, and `review-ai`.
- Mandatory Workflow: read memory and board before editing, declare identity before acting, update memory after substantive changes, update the board for collaboration or blockers.
- Test Gate: list test, lint, build, smoke, E2E, and lab validation requirements according to the actual tech stack.
- Language: specify language requirements for code, logs, config, protocol fields, and user-facing docs.

`AGENTS.md` is only a summary and navigation file. Do not copy detailed standards into it.

### `MEMORY.md`

`MEMORY.md` is current project memory, not a complete historical encyclopedia. It must include:

```md
# MEMORY

Last updated: YYYY-MM-DD HH:MM <timezone>

## Current Phase

## Completed Items

## AI Engineering Status Checklist

## Follow-up Suggestions

## Pending Confirmation

## Change Log
```

Requirements:

- The current phase must be based on the real project state.
- Prefer keeping it under 100 lines. Archive old history to `.ai/docs/changelog.md` or `.ai/docs/archive/` when it grows too long.
- Change log entries use one line:

```text
- YYYY-MM-DD HH:MM | <model-name> | <one-sentence summary of this change>
```

- Use local timezone and minute precision.
- Keep descriptions under 80 characters.
- Mark uncertain content with `<!-- TBD -->`.

### `AI_BOARD.md`

`AI_BOARD.md` is the only active collaboration board. It must include:

```md
# AI Collaboration Board

## File Model

## AI Identities

## Active Board

## Board Rules

## Activity Log Format

## Activity Log
```

Active Board table format:

```md
| ID | Owner | Type | Status | Item | Next Action |
| --- | --- | --- | --- | --- | --- |
```

Valid identities:

| Identity | Owner | Scope | Boundary |
| --- | --- | --- | --- |
| Frontend AI | `frontend-ai` | Frontend UI, Wails/IPC call wrappers, trilingual copy, visuals, and interactions. | Does not implement sync core; does not directly access RabbitMQ/MySQL/HTTP. |
| Backend AI | `backend-ai` | Backend, services, sync core, config, DTOs, installers, MCP. | Frontend-visible API/DTO/error semantic changes must be written to the board. |
| Test AI | `test-ai` | Unit tests, smoke, E2E, lab, load tests, release-gate evidence. | Does not change product behavior by default; test fixture fixes must state scope. |
| Review AI | `review-ai` | Architecture review, launch assessment, risk list, cross-module adjustments. | List risks and gaps first; cross-identity edits must state reason and impact. |

Board Rules must include:

1. Declare identity before starting, then read Active Board.
2. New issues must be added to Active Board and assigned a valid `Owner`.
3. When resolved, change status to `closed` and append Activity Log.
4. When blocked, change status to `blocked` and state what is missing.
5. For API, DTO, error semantics, or page-scope changes, update Active Board first, then sync stable contract docs.
6. `test-ai` must record validation commands, pass/fail results, and reasons for skipped runs.
7. `review-ai` must list risks, gaps, and blockers first.
8. Final replies must state identity, handled Board items, and remaining open/blocked items.
9. When closed history grows too long, `review-ai` archives it to `.ai/docs/archive/` or a phase summary.

Activity Log format:

```text
- YYYY-MM-DD HH:mm | <frontend-ai/backend-ai/test-ai/review-ai> | <question/decision/answer/blocker/review/test> | <scope> | <open/closed/blocked>
```

## Step 5: Create or Maintain the `.ai/` Source-of-Truth Documentation System

Required directories:

```text
.ai/
  instructions/
  docs/
    archive/
  skills/
  agents/
  prompts/
```

Every directory must have a `README.md` explaining:

- Directory purpose
- File naming conventions
- When to create new files
- When not to create new files

Required files:

```text
.ai/instructions/ai-workflow.md
.ai/docs/README.md
.ai/skills/README.md
.ai/agents/README.md
.ai/prompts/README.md
```

Create as needed:

```text
.ai/instructions/<domain>.md
.ai/docs/<topic>.md
.ai/skills/<skill-name>/SKILL.md
.ai/agents/<name>.agent.md
.ai/prompts/<name>.prompt.md
```

As-needed files must be backed by project evidence. Do not create hollow files just to fill structure.

### `.ai/instructions/*.md`

Must have frontmatter:

```yaml
---
description: "Use when: trigger keyword description"
applyTo: "glob pattern"
---
```

Requirements:

- One file handles one concern, such as backend, frontend, API, testing, or sync architecture.
- Keep content short and actionable.
- Do not copy the project summary from `AGENTS.md`.
- Must explain the relationship with `MEMORY.md` and `AI_BOARD.md`.
- Frontend, backend, test, or review collaboration instructions must require reading root `AI_BOARD.md`.

### `.ai/docs/*.md`

Purpose:

- Stable design documents
- Architecture notes
- API contracts
- Phase summaries
- Closed records
- Archive materials

Prohibited:

- Do not create an active board under `.ai/docs/`.
- Do not maintain open/blocked work items under `.ai/docs/`.
- Move invalidated content into `.ai/docs/archive/`.

### `.ai/skills/<skill-name>/SKILL.md`

Must have frontmatter:

```yaml
---
name: skill-name
description: "Use when: keyword-rich trigger description"
argument-hint: "input hint"
---
```

Create a skill only when all conditions are met:

- The project has a repeated operation that appears 3+ times.
- Inputs and outputs are clear.
- Steps are stable and reproducible.
- It contains project-specific constraints.
- It can be reused for future similar tasks.

Do not create generic hollow skills such as "read code" or "fix bug".

### `.ai/agents/<name>.agent.md`

Must have frontmatter:

```yaml
---
description: "Use when: trigger description"
name: "agent-name"
tools: [read, search]
---
```

Requirements:

- Create at least one read-only review Agent.
- Define role, responsibilities, boundaries, and what it does not do.
- Use the minimum necessary tool set.
- Review Agents are read-only by default.
- If an Agent relates to the identity model, it must correspond to `frontend-ai`, `backend-ai`, `test-ai`, or `review-ai`.

### `.ai/prompts/<name>.prompt.md`

Must have frontmatter:

```yaml
---
description: "One-sentence purpose"
agent: "agent"
tools: [read, edit, search]
---
```

Requirements:

- Use Markdown links to reference related instructions / skills / agents.
- Create prompts for the project's 3 to 5 most common operations.
- Do not create low-value prompts just to fill a quota.
- If the project has few scenarios, fewer than 3 prompts is acceptable.
- Identity entry prompts should require declaring identity first and reading `AI_BOARD.md`.

## Step 6: Deep-Review Workflow

Do not create the deep-review workflow by default. Create it incrementally only when the user explicitly says "enable review workflow", "add deep review", `/review`, "create deep-review Agent", or equivalent trigger phrases:

```text
.ai/agents/deep-review.agent.md
.ai/prompts/deep-review.prompt.md
tests/README.md or an ai-review note under the existing test directory
## Review Log section in MEMORY.md
```

The deep-review Agent's minimum tools are `[read, search, edit, run]`, and its loop is fixed:

1. Review: output issue list based on `.ai/instructions/`, `AI_BOARD.md`, and recent diff.
2. Add tests: add tests for uncovered critical paths; do not write empty tests.
3. Run tests: run project test commands and record failures.
4. Fix failures: minimally fix failures; never delete or skip tests to manufacture a pass.
5. Review again: return to review until two consecutive rounds have no new issues and all tests pass.

At the end of each round, append to `MEMORY.md` `## Review Log`:

```text
- YYYY-MM-DD HH:MM | <model-name> | round <N> | <one-sentence summary and conclusion: pass / continue next round>
```

## Step 7: Editor Adapter Layers

`.ai/` is the source of truth. Editor adapter layers are thin entries only and must not copy full standards.

### Copilot

Create or maintain:

```text
.github/copilot-instructions.md
.github/instructions/*.instructions.md
```

`.github/copilot-instructions.md`:

- 10 to 20 lines.
- Only write the core entry.
- Must require reading `AGENTS.md` first.
- Must mention `MEMORY.md` and `AI_BOARD.md`.
- Must not copy full standards.

`.github/instructions/*.instructions.md`:

- Create only 2 to 4 adapters for core `.ai/instructions/*.md`.
- Must have frontmatter:

```yaml
---
applyTo: "glob pattern"
---
```

- Body only states which source files to read and 2 to 4 hard rules.

### Cursor

Create or maintain:

```text
.cursor/rules/00-core.mdc
.cursor/rules/<domain>.mdc
```

`00-core.mdc`:

- `alwaysApply: true`
- `globs: ["**/*"]`
- Only write the entry, not full standards.

Domain rules:

- `alwaysApply: false`
- `globs` matches the corresponding `.ai/instructions/*.md` `applyTo`.
- Only write source links and a few hard rules.

### Codex

Codex natively reads root `AGENTS.md`; no extra adapter file is needed.

### Trae

Trae reuses `AGENTS.md` and the `.ai/` source by default. Create a thin Trae entry only if the project already has a native Trae rules directory or the user explicitly asks for one.

Trae adapter rules:

- If a native Trae rules directory exists, preserve its structure and add only source links.
- If the Trae rules path cannot be confirmed, mark `<!-- TBD -->` and do not invent a directory.
- The Trae adapter layer must not copy full standards.
- The Trae adapter layer must not create a second active board.

## Step 8: Old Board Migration

If an old active board is found at:

```text
.ai/docs/ai-collaboration-log.md
```

Handle it as follows:

- Migrate still-valid open/blocked items into root `AI_BOARD.md`.
- Keep only a migration notice in the old file.
- State clearly: do not add new open/blocked items to the old file.
- Update all guides to point to root `AI_BOARD.md`.

Old file content example:

```md
# AI Collaboration Board moved

This file is no longer the active board. The active AI collaboration board has moved to the repository root:

AI_BOARD.md

Rules:
- `AI_BOARD.md` is the only active board.
- `.ai/docs/` only stores stable docs, closed records, and archive materials.
- Do not add open/blocked items to this file.
```

## Step 9: Safe Write Rules

- Prefer merging existing files. Do not blindly overwrite.
- If original content conflicts with current analysis, preserve it and mark `<!-- TBD -->`.
- Uncertain content must be explicitly marked `<!-- TBD -->`.
- If similar docs already exist, link to them instead of duplicating large sections.
- Do not create empty business documents.
- Do not use build artifacts, caches, or generated code as current standards evidence.
- When paths change, sync all references.
- Adapter layers must always be shorter than the source files.
- Do not create an active board that competes with `AI_BOARD.md`.

## Step 10: Self-Check After Completion

Format checks:

- `AGENTS.md` exists and is under 300 lines.
- `MEMORY.md` exists and has a last-updated timestamp.
- `AI_BOARD.md` exists and is the only active board.
- `.ai/` directory structure is complete.
- Every README explains purpose and creation criteria.
- instructions / skills / agents / prompts have correct frontmatter.
- A skill's `name` matches its directory name.
- descriptions use `Use when: ...`.
- `.ai/docs/` has no active board.
- `.ai/docs/ai-collaboration-log.md`, if present, is only a migration notice.
- Copilot / Cursor / Trae adapter layers are entries only.
- No references point to missing files.

Content credibility checks:

- Every instruction has project evidence.
- Every skill corresponds to a real high-frequency workflow.
- Every prompt references an existing instruction, skill, or agent.
- Module breakdown matches actual directory structure and entry files.
- All unconfirmed content is marked `<!-- TBD -->`.
- Build artifacts, generated code, and legacy leftovers are not used as current standards evidence.

Collaboration checks:

- Active Board owners use only valid identities.
- Active Board open/blocked items are clear.
- Activity Log format is consistent.
- Final reply states identity, handled Board items, and remaining open/blocked items.
- Substantive changes have been written to `MEMORY.md`.
- Collaboration, blockers, API, DTO, error semantics, or page-scope changes have updated `AI_BOARD.md`.

Deep-review checks:

- When deep review is not triggered, do not create `deep-review.agent.md`, `deep-review.prompt.md`, or review test directories.
- When deep review is triggered, the review Agent, Prompt, test note, and `MEMORY.md ## Review Log` are all synced.

## Completion Output

Finally output:

1. Judged project state: new / partially initialized / initialized.
2. Created or updated file list.
3. Current `AI_BOARD.md` open/blocked items.
4. Self-check summary: passed and failed items.
5. Pending confirmation summary.
6. Follow-up suggestions.
