あなたはAIエンジニアリングの専門家です。現在のプロジェクトに完全なAI協業ドキュメント体系を構築してください。

## 目標

プロジェクト内に、AIエージェント（エディタ横断：Copilot、Cursor、Trae等）が効率的に協業できるドキュメント構造を確立する。含む内容：プロジェクトガイド、コーディング規約、再利用可能なスキル、カスタムエージェント、プロンプトテンプレート、開発進捗管理。

## 要件

### ステップ1：プロジェクト分析

#### 1.1 スキャン境界

まず分析範囲を明確にする：
- **無視するディレクトリ**：`node_modules`、`dist`、`build`、`.next`、`coverage`、`vendor`、`generated`、`__pycache__`、`.cache`、ロックファイル等のビルド成果物/依存関係/キャッシュ/生成ファイル
- **優先読取**：ルートREADME、パッケージ管理設定（package.json / go.mod / Cargo.toml / pom.xml等）、エントリファイル、コアビジネスディレクトリ、テストディレクトリ、CI設定
- **大規模プロジェクト戦略**：プロジェクトのコアモジュールが20を超える場合やコード量が膨大な場合、まずコアモジュールをサンプリング分析し、全量スキャンは求めない。残りのモジュールはバックログに入れて後で対応- **コアモジュールの判定**：エントリポイント参照チェーン、主要ビジネスディレクトリ、ビルド設定、CIカバレッジ範囲、テストカバレッジ密度、READMEで明示されたモジュールを総合的に判断
- **"コード量が覆大"の判定**：明らかなマルチサブシステム特徴、複雑なディレクトリ階層、分散した設定とエントリポイント、または1回の分析で信頼できるグローバル認知が形成できない状況を指す
#### 1.2 分析内容

スキャン境界内で以下を把握する：
- プロジェクトの位置づけとコア責務
- 技術スタック（言語、フレームワーク、ビルドツール）
- モジュール分割と各責務
- 既存のコーディング規約とアーキテクチャ上の決定
- プロジェクトの現在フェーズ（初期構築/開発中/保守）
- 繰り返し出現する開発オペレーションパターン

#### 1.3 情報ソース優先順位

異なるソースの情報が矛盾する場合、以下の優先順位で判断する：

**実行設定 / ビルドスクリプト / CI設定 / 現在のソース構造 > テストケース > README > その他の履歴ドキュメント > コメント > issue / commit**

#### 1.4 デグレード戦略

分析中に不確実な状況に遭遇した場合：
- プロジェクト規約に一貫性がない場合、統一を強制せず「現状」を記録する
- アーキテクチャ決定が確認できない場合、`<!-- 要確認 -->` と標記する
- プロジェクトが大きすぎて全量分析できない場合、コアモジュールのみカバーし、残りをバックログに
- 既存規約が矛盾する場合、矛盾の原因と各方の現状を説明し、独断しない

### ステップ2：分析レポートの出力

**すぐにファイルを作成しないこと。** まず以下を出力して確認を待つ：
1. プロジェクト認知サマリー（位置づけ、技術スタック、モジュール、フェーズ、コア規約）
2. 提案する `.ai/` ディレクトリ構造とファイルリスト
3. 各ファイルの想定内容サマリー（一文）
4. 特定された不確実点と要確認事項
5. 大規模プロジェクトの場合、今回のカバー範囲と残りのバックログ

各重要判断には少なくとも1つの根拠ソース説明（例：どのディレクトリ、設定ファイル、テスト、エントリファイルか）を付記する。詳細な引用は不要だが、追跡可能であること。

ユーザー確認後にステップ3へ進む。ユーザーが「そのまま作成して」と明示した場合、またはプロジェクトが以下の多くの条件を満たす場合、確認をスキップして直接実行可：
- コアモジュール ≤10
- 技術スタックが2〜3カテゴリ以内
- ルートディレクトリ構造が明確で、明らかなマルチサブシステム / monorepo特徴がない
- 既存ドキュメントとコードが概ね一致している

### ステップ3：優先順位に従ってファイル作成

#### 3.1 出力優先順位

以下の順序で作成し、コアファイルの品質を優先する：

**第一優先（必須）：**
1. `AGENTS.md` — プロジェクト概要とナビゲーションエントリ（自動ロード、ガイド兼任）
2. `.ai/instructions/ai-workflow.md` — ドキュメント更新規約
3. コアinstructions（2〜4個の高価値規約ファイル）
4. `MEMORY.md` — 開発進捗管理

**第二優先（必要に応じて）：**
5. `.ai/docs/` — 機能設計ドキュメント
6. `.ai/skills/` — 高頻度スキル（2〜3個）
7. `.ai/agents/` — 最低1つの読み取り専用レビューエージェント
8. `.ai/prompts/` — よく使うプロンプトテンプレート

**第三優先（エディタアダプタ）：**
9. `.github/copilot-instructions.md` — Copilotリポジトリレベルエントリ
10. `.github/instructions/*.instructions.md` — Copilotパスレベルルール
11. `.cursor/rules/*.mdc` — Cursorルールエントリ

#### 3.2 安全書き込みルール

- 対象ファイルが既に存在する場合、盲目的に上書きせず**マージ・補完**を優先。マージ時は元ファイルの構造を保持し、不足セクションを補完する。元の内容が現在の分析と矛盾する場合、元のテキストを保持し `<!-- 要確認 -->` と標記
- 不確実な内容は `<!-- 要確認 -->` と記載し、確定した結論に偽装しない
- 具体的なskill / agent / prompt / docファイルは必要に応じて作成するが、**ディレクトリ構造は必ず全て作成**する
- 各オンデマンドディレクトリ（docs / skills / agents / prompts）には `README.md` を必ず配置。ディレクトリの用途、ファイル命名規約、作成条件を説明し、ユーザーの混乱を防ぐ
- ディレクトリREADME以外に、構造を埋めるためだけのプレースホルダー型業務ファイルを作成しない。オンデマンドファイルはプロジェクトの証拠がある場合のみ作成
- プロジェクトに同種のドキュメントが既にある場合、重複記述せず**既存ドキュメントへリンク**する

#### 3.3 ファイル構造

全ディレクトリを必ず作成する。`[随時]` と記載のファイルはプロジェクトの実状に応じて作成を判断。

```
AGENTS.md                     — プロジェクト概要とナビゲーションエントリ（ルートディレクトリ、≤300行、自動ロード）
MEMORY.md                     — 開発進捗管理（ルートディレクトリ）
.ai/
  instructions/               — 分野別コーディング規約
    ai-workflow.md            — ドキュメント更新規約（必須）
    *.md                      — その他の規約ファイル（随時）
  docs/                       — 機能設計・追跡ドキュメント
    README.md                 — ディレクトリ説明：用途、ファイル命名規約、作成条件
    changelog.md              — MEMORY.md履歴アーカイブインデックス（随時）
    archive/                  — 失効したprompt / skill / docのアーカイブディレクトリ（随時）
    *.md                      — 設計ドキュメント（随時）
  skills/                     — 再利用可能なオペレーションテンプレート
    README.md                 — ディレクトリ説明：スキルとは、抽出基準、構造要件
    <skill-name>/SKILL.md     — 具体的なスキル（随時）
  agents/                     — カスタムエージェント
    README.md                 — ディレクトリ説明：エージェント定義形式、ツール制限ガイド
    <name>.agent.md           — 具体的なエージェント（随時）
  prompts/                    — プロンプトショートカットテンプレート
    README.md                 — ディレクトリ説明：プロンプト形式要件、参照方法
    <name>.prompt.md          — 具体的なプロンプト（随時）
```

> 深度レビューフロー関連成果物（`tests/`、`deep-review.agent.md`、`deep-review.prompt.md`、`MEMORY.md` の `## レビューログ` セクション）は**ユーザーが主体的に発起する**オプション機能であり、**デフォルトでは作成せず、能動的に確認もしない**。ユーザーが明示的に要求した場合（「レビューフロー有効化」「深度レビュー追加」「/review」等のトリガー）にのみ、本プロンプトの「深度レビューフロー成果物」節に従って増分で作成する。

### ステップ4：各ファイルのフォーマット要件

#### AGENTS.md（ルートディレクトリ — 唯一の自動ロードエントリ）

AGENTS.mdはエディタ（Copilot / Cursor等）が毎回の会話で自動ロードするファイルであるため、「プロジェクト概要」と「ナビゲーションガイド」の両方の役割を担う。独立した `.ai-instructions` ガイドファイルは作成しない。

- タイトル：`# Project Guidelines`

**上半分——プロジェクトサマリー：**
- プロジェクト位置づけ（1-2文）
- 技術スタック（リスト）
- コアモジュール一覧（テーブル、コアモジュールのみ記載、非コアは省略）
- コア規約（4-5項目）

**下半分——ドキュメントナビゲーション：**
- **必読ファイル**：重要度順にMEMORY.md → コアinstructions（最も重要な2〜3個のみ、一行説明付き）を記載
- **オンデマンドリソース**：`.ai/` 下のdocs / skills / agents / promptsのキーファイルとトリガー条件の精簡テーブル（YAMLフロントマターのdescriptionが発見を補助するため、ここでは最も頻繁に使用される項目のみ記載）
- **強制ワークフロー**：開始前に何を読むか → 完了後にMEMORY.mdを更新
- **言語要件**

詳細は `.ai/docs/` に分割、AGENTS.mdはインデックスとナビゲーションのみ。300行以内に収める — 毎回の会話で全量ロードされるため。

#### .ai/instructions/*.md（コーディング規約）
- 各ファイルにYAMLフロントマターが必須：
  ```yaml
  ---
  description: "Use when: トリガーキーワードの説明"
  applyTo: "globパターン"    # 任意、ファイルタイプで自動アタッチ
  ---
  ```
- 1ファイル1関心事（バックエンド/フロントエンド/API/テスト — 混在禁止）
- 内容は簡潔で実行可能に。長文ではなくリストで
- 多言語プロジェクトの場合、言語またはドメインで分割
- `ai-workflow.md` を必ず作成し、ドキュメント更新規約を定義

#### .ai/skills/<name>/SKILL.md（スキルテンプレート）
- YAMLフロントマター必須：
  ```yaml
  ---
  name: skill-name           # ディレクトリ名と一致必須
  description: "Use when: キーワード豊富なトリガー説明"
  argument-hint: "入力ヒント"
  ---
  ```
- 本文に含める：適用シナリオ、プロジェクト規約（コード例付き）、ステップバイステップ手順、重要な制約
- **スキル抽出基準**：以下の全条件を満たす場合のみスキルとして抽出：
  - プロジェクトで3回以上出現する繰り返しパターン
  - 明確な入力と出力がある
  - ステップが安定して再現可能
  - プロジェクト固有の制約や規約を含む
  - 将来の類似タスクに再利用可能
- プロジェクト固有の特徴がない汎用操作（例：「コードを読む」「バグを直す」）はスキルにしない

#### .ai/agents/<name>.agent.md（カスタムエージェント）
- YAMLフロントマター必須：
  ```yaml
  ---
  description: "Use when: トリガー説明"
  name: "agent-name"
  tools: [read, search]      # 最小ツールセット
  ---
  ```
- 本文で定義：役割位置づけ、責務範囲、制約（やらないこと）、具体的なチェックリストまたは作業手順
- 読み取り専用レビューエージェント（tools: [read, search]）を最低1つ作成
- `tools` はドキュメントレベルの能力制約宣言であり、最小必要能力を表現するもので、特定のエディタ実装に依存しない

#### .ai/prompts/<name>.prompt.md（プロンプトテンプレート）
- YAMLフロントマター必須：
  ```yaml
  ---
  description: "一行で用途を説明"
  agent: "agent"
  tools: [read, edit, search]
  ---
  ```
- 本文でMarkdownリンクを使い、関連するinstructionsとSKILL.mdを参照
- プロジェクトで最も頻繁な3〜5の操作に対してプロンプトを作成。プロジェクトのシナリオが少ない場合3個未満でも可だが、数合わせのために低価値テンプレートを作成しない

#### MEMORY.md（開発進捗）
- 含める：現在フェーズ、完了事項、AIエンジニアリング状態チェックリスト、今後の提案、**変更履歴**
- ファイルヘッダーに最終更新日を記載
- 100行以内に収める。超過分は `.ai/docs/changelog.md` にアーカイブ
- **変更履歴フォーマット**：1件1行、簡潔明瞭、モデル名と分単位精度のタイムスタンプを必ず含める
  ```
  - YYYY-MM-DD HH:MM | <model-name> | <一文で本回の変更を説明>
  ```
  - タイムゾーンはローカル、分単位（日付のみは不可）
  - `model-name` は本回の変更を実行したモデル識別子。例：`claude-opus-4.7`、`gpt-5`、`gemini-2.5-pro`
  - 説明は80文字以内、複数段落の長文禁止。複数ファイルの変更は1件にまとめる
- 深度レビューフロー有効時は別途 `## レビューログ` セクションを設け、各ラウンド1行で記録（下記の深度レビューセクション参照）

#### 深度レビューフロー成果物（ユーザーが主体的に発起した場合のみ作成、デフォルトでは作成しない）

##### tests/ ディレクトリ
- プロジェクトに既存のテストディレクトリとフレームワークがあれば優先再利用。なければ言語慣例で新規作成
- `tests/README.md` には：使用テストフレームワーク、統一実行コマンド、ディレクトリ組織、期待カバレッジを必ず記載
- `ai-review/`（または等価サブディレクトリ）にはレビューフロー由来の回帰テストを格納。ファイル名はトリガーとなったレビュー項目に追跡可能であること

##### .ai/agents/deep-review.agent.md
- frontmatter `tools` の最小セット：`[read, search, edit, run]`（テスト実行のため `run` 必須）
- 5ステップループ固定、必ず順序通り実行：
  1. **レビュー**：`.ai/instructions/` と最新diffに基づき問題リストを出力
  2. **テスト追加**：未カバーのクリティカルパスに対し `tests/` にテスト追加
  3. **テスト実行**：プロジェクトのテストコマンドを実行し失敗を収集
  4. **失敗修正**：失敗に対する最小限の修正。**テストの削除/スキップで失敗を回避することは禁止**
  5. **再レビュー**：ステップ1に戻り、新たな問題がなくテストが全て通るまで繰り返す
- 終了条件：連続2ラウンドで新規問題なし、かつテスト全通過
- 各ラウンド終了時に `MEMORY.md` の `## レビューログ` セクションへ1件追記必須

##### .ai/prompts/deep-review.prompt.md
- `deep-review.agent.md` および関連instructionsを参照
- 起動エントリを提供（例：`/review` または一行トリガー）

##### MEMORY.md レビューログフォーマット
```
## レビューログ
- YYYY-MM-DD HH:MM | <model-name> | round <N> | <一文で本ラウンドの変更と結論>
```
- 分単位精度。`round N` は現在のループ回数。結論は「合格 / 次ラウンドへ継続」を明示すること

### ステップ5：コンテンツ品質チェックリスト

作成後のセルフチェック：

#### 形式チェック
- [ ] AGENTS.md ≤ 300行、上半分はプロジェクトサマリー、下半分はナビゲーションガイド
- [ ] すべてのinstructions/skills/agents/promptsに正しいYAMLフロントマターあり
- [ ] skillsの `name` フィールドがディレクトリ名と一致
- [ ] すべてのdescriptionが "Use when: ..." 形式でキーワード豊富
- [ ] AGENTS.mdナビゲーション区に記載の全ファイルが実際に存在
- [ ] MEMORY.mdが存在しないファイルを参照していない
- [ ] instructions間でコンテンツの重複なし（特にAGENTS.mdとの間）
- [ ] 各エージェントのtoolsが最小必要セット（全ツールではない）
- [ ] 強制ワークフローで完了後のMEMORY.md更新を要求
- [ ] MEMORY.mdの変更履歴フォーマット遵守（モデル名 + 分単位タイムスタンプ、1行簡潔）
- [ ] デフォルトでは深度レビュー成果物を作成しない。ユーザーが主体的に発起した場合のみ、`tests/`、`deep-review.agent.md`、`deep-review.prompt.md`、`MEMORY.md` の `## レビューログ` がすべて作成済み

#### コンテンツ信頼性チェック
- [ ] 各instructionがプロジェクト内で少なくとも1つの証拠ソース（コード、設定、テスト）に紐づく
- [ ] 各skillが少なくとも1つの実在する高頻度開発フローに対応
- [ ] 各promptが実際に存在するinstructionまたはskillを参照
- [ ] モジュール分割がプロジェクトの実際のディレクトリ構造とエントリファイルに一致
- [ ] 未確認コンテンツがすべて `<!-- 要確認 -->` と明示され、確定事実に偽装していない
- [ ] ビルド成果物、生成コード、レガシー残存物を現在の規約の根拠にしていない

#### アダプタレイヤーチェック
- [ ] アダプタファイルに完全な規約本文が含まれていない—ナビゲーションとエントリのみ
- [ ] アダプタファイルが参照する `.ai/` ソースファイルが実際に存在
- [ ] `.github/instructions/*.instructions.md` の `applyTo` が対応する `.ai/instructions/*.md` のカバレッジと一致
- [ ] `.cursor/rules/*.mdc` の `globs` が対応する `.ai/instructions/*.md` のカバレッジと一致
- [ ] 各アダプタファイルの行数が対応するソースファイルを超えていない

### 完了検収

作成完了後、以下を出力して本ラウンドを終了する：
1. 作成済みファイルリスト（パス付き）
2. セルフチェック結果サマリー（合格 / 不合格項目）
3. 要確認項目のまとめ
4. 今後の提案（バックログや補充事項がある場合）

### メンテナンスと進化

`.ai/` 体系が陳腐化しないよう、以下のルールを遵守：
- `ai-workflow.md` で定義：どのプロジェクト変更がinstructions更新をトリガーするか（例：技術スタック移行、ディレクトリ再構成、新モジュール追加）
- 無効化されたpromptやskillは速やかに `.ai/docs/archive/` にアーカイブ。アクティブディレクトリに残さない
- MEMORY.mdが100行を超えたら `.ai/docs/changelog.md` にアーカイブ
- 大規模リファクタリング後は `.ai/` 体系の正確性を再評価

### 大規模プロジェクト適応

プロジェクトがmonorepoまたは複数サブシステムを含む場合：
- ルートレベルの `.ai/` に汎用規約とナビゲーションを格納
- 各サブプロジェクト/サブモジュールは独自のローカル `.ai/instructions/` を持てる
- ルートディレクトリにはサブシステム横断の共通規約とナビゲーションインデックスのみ保持
- サブシステム固有の規約をルートレベルのinstructionsに詰め込まない
- ルートレベル規約とサブシステムローカル規約が矛盾する場合、サブシステムローカル規約がそのスコープ内で優先。ルートレベル規約は共通制約のみ提供

### エディタアダプタレイヤー

`.ai/` 体系がソースオブトゥルースであり、全規約内容を格納する。CopilotとCursorのネイティブルール発見能力を向上させるため、薄いアダプタエントリレイヤーを補完する。

#### 総原則

- `.ai/` に全量コンテンツを格納、アダプタレイヤーは短いエントリのみ—規約の複製禁止
- アダプタファイルの役割は「エディタにどのソースファイルを読むか伝える」こと—ルールの複製ではない
- アダプタファイルは常にソースより短く、1ファイル10〜20行以内
- ソースのみ修正、アダプタレイヤーはソースファイルのリネームやパス変更時のみ参照を同期

#### Copilotアダプタ（必須）

Copilotは `.github/copilot-instructions.md`（リポジトリレベル）と `.github/instructions/*.instructions.md`（パスレベル）をネイティブに認識する。

**`.github/copilot-instructions.md`**：
- 10〜20行以内
- 内容：まずAGENTS.mdを読む（プロジェクト概要とナビゲーションガイドが含まれている）、情報ソース優先順位、要確認マークルール、作業後にMEMORY.md更新、言語要件
- 具体的なコーディング規約は含めない

**`.github/instructions/*.instructions.md`**：
- コア `.ai/instructions/*.md` のみ対応アダプタを作成（2〜4ファイル）—全量ミラー禁止
- ファイル名は `.instructions.md` で終わること
- YAMLフロントマターに `applyTo` 必須、対応ソースの `applyTo` カバレッジと一致
- 本文：どのソースファイルを読むか + 2〜4条のキー制約のみ

例：
```md
---
applyTo: "**/*.ts,**/*.tsx,src/frontend/**"
---

Read `AGENTS.md` and `.ai/instructions/frontend.md` before editing these files.

Rules:
- Follow existing component patterns before introducing new abstractions.
- Keep changes minimal and consistent with nearby files.
- Update related tests when behavior changes.
- Mark uncertain decisions as `<!-- 要確認 -->`.
```

#### Cursorアダプタ（推奨）

Cursorは `.cursor/rules/*.mdc` ディレクトリ下のルールファイルを認識する。

**`.cursor/rules/00-core.mdc`**：
- `alwaysApply: true`、`globs: ["**/*"]` を設定
- 内容は `copilot-instructions.md` と同等：まずAGENTS.mdを読む（ナビゲーションガイドが含まれている）、同じコア制約

**`.cursor/rules/<domain>.mdc`**：
- コアドメインのみ作成（2〜4ファイル）—全量ミラー禁止
- `alwaysApply: false`、`globs` で対応ソースの `applyTo` と一致する範囲を指定
- 本文：どのソースファイルを読むか + 2〜4条のキー制約のみ

例：
```md
---
description: Frontend file editing rules
globs: ["**/*.tsx", "**/*.jsx", "src/frontend/**"]
alwaysApply: false
---

Before editing these files, read:
- `AGENTS.md`
- `.ai/instructions/frontend.md`

Rules:
- Reuse existing UI patterns before creating new abstractions.
- Keep props and state handling consistent with nearby files.
- Update related tests when UI behavior changes.
```

#### アダプタマッピング

| ソース | Copilotアダプタ | Cursorアダプタ |
|----------|------------------|-----------------|
| `AGENTS.md`（概要+ナビゲーション） | そのまま再利用 + `.github/copilot-instructions.md` で参照 | そのまま再利用 + `00-core.mdc` で参照 |
| `.ai/instructions/<domain>.md` | `.github/instructions/<domain>.instructions.md` | `.cursor/rules/<domain>.mdc` |
| `MEMORY.md` | リポジトリレベルエントリで更新要求 | コアルールで更新要求 |

#### 同期規律

1. **ソースのみ修正**：規約変更はまず `.ai/` へ—アダプタレイヤーを直接修正しない
2. **参照同期**：ソースファイルのリネームやパス変更時のみアダプタの参照を同期
3. **詳細禁止**：アダプタはナビゲーション、トリガー、最小限のハードルールのみ

### 普遍原則

- **証拠優先**：ドキュメント内容はソースコード、ビルド設定、テスト、CIに基づくこと。捏造禁止
- **簡潔第一**：AGENTS.mdとinstructionsはコンテキストウィンドウを消費する — 短いほど良い
- **キーワード駆動の発見**：descriptionがエージェントのファイル発見の唯一根拠 — キーワードは正確に
- **1ファイル1関心事**：テスト規約とAPI規約を混在させない
- **見せて語る**：長文説明より簡潔なコード例を
- **リンクして埋め込まない**：詳細はdocsに、メインファイルはインデックスのみ
- **エディタ横断互換**：特定エディタの仕組み（Hooks等）に依存しない。ファイル形式は汎用に
- **安全な書き込み**：既存ファイルは上書きより補完を優先。不確実な点は要確認と標記。空ファイル作成禁止
