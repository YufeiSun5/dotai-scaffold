# AI協業ドキュメント体系 初期化プロンプト

あなたはAIエンジニアリングの専門家です。現在のプロジェクトに、完全なAI協業ドキュメント体系を初期化または保守してください。

## 目標

Copilot、Cursor、Trae、Codexなどの一般的なツールに対応し、AI Agentがエディタ横断で協業できるドキュメント構造をプロジェクト内に構築する。この体系は必ず以下を含む：

- プロジェクト案内入口
- 現在のプロジェクトメモリ
- アクティブな協業ボード
- 分野別コーディング規約
- 安定した設計文書とアーカイブ文書
- 再利用可能なスキル
- カスタムAgent
- Promptテンプレート
- エディタ用の薄いアダプタ層

## コアファイルモデル

必ず以下のファイル境界を採用する：

```text
AGENTS.md                     # プロジェクト案内と自動ロード入口
MEMORY.md                     # 現在フェーズ、完了事項、要確認事項、変更履歴
AI_BOARD.md                   # 唯一のアクティブAI協業ボード
.ai/                          # AI母本文書体系
  instructions/               # 分野別規約
  docs/                       # 安定文書、完了記録、フェーズまとめ、アーカイブ
  skills/                     # 再利用可能なスキル
  agents/                     # カスタムAgent
  prompts/                    # Promptテンプレート
.github/                      # Copilot用の薄いアダプタ層
.cursor/                      # Cursor用の薄いアダプタ層
```

`AI_BOARD.md` は `MEMORY.md` と同階層に置き、唯一のアクティブ協業ボードでなければならない。`.ai/docs/` には安定文書、完了記録、フェーズまとめ、アーカイブ資料のみを置く。open/blockedの作業項目を載せてはならず、第二のアクティブボードを作ってはならない。

## 実行モード

デフォルトは二段階実行とする。第一段階ではプロジェクトをスキャンし分析レポートを出力するだけで、いかなるファイルも作成・変更しない。ユーザーがレポートを確認した後、第二段階として初期化または保守に進む。

初回リクエストに「初期化」「作成」「生成」と書かれていても、必ず先にレポートを出力し、二回目の確認を待つ。ユーザーがレポートを読んだ後に「確認」「初期化を開始」「レポート通りに作成」「続行」などの確認語を明示した場合のみ、ファイル書き込みを許可する。

ユーザーが「分析のみ」「先にレポート」「読み取り専用モード」と明示した場合は、分析レポートのみを出力し、ファイルの作成・変更を行わず、初期化にも能動的に進まない。

確認後に実行する際、`MEMORY.md` または `AI_BOARD.md` が存在しなくても停止しない。プロジェクトを新規または半初期化状態と判断し、まずルート外側の協業ファイルを作成し、その後 `.ai/` 母本文書体系を作成する。

確認後に実行する際、既存ファイルがある場合はマージと補完を優先し、盲目的に上書きしない。衝突する元内容は保持し、`<!-- 要確認 -->` と標記する。

## ステップ1：プロジェクトをスキャンする

まずスキャン境界を明確にする。

無視するディレクトリとファイル：

- `node_modules`
- `dist`
- `build`
- `.next`
- `coverage`
- `vendor`
- `generated`
- `__pycache__`
- `.cache`
- ロックファイル
- ビルド成果物、依存ディレクトリ、実行キャッシュ、生成コード

優先して読む：

- `AGENTS.md`
- `MEMORY.md`
- `AI_BOARD.md`
- ルートREADME
- パッケージ/ビルド設定：`package.json` / `go.mod` / `Cargo.toml` / `pom.xml` など
- エントリファイル
- コア業務ディレクトリ
- テストディレクトリ
- CI設定とビルドスクリプト

分析する内容：

- プロジェクトの位置づけとコア責務
- 技術スタック、フレームワーク、ビルドツール
- モジュール分割と責務
- 既存のコーディング規約とアーキテクチャ判断
- 現在フェーズ：初期構築、開発中、試験導入、保守、納品前など
- 高頻度の開発オペレーションパターン
- 既存のAI協業文書が古い、重複している、または衝突しているか

情報源の優先順位：

```text
実行設定 / ビルドスクリプト / CI設定 / 現在のソース構造 > テストケース > README > その他の履歴文書 > コメント > issue / commit
```

大規模プロジェクト戦略：

- コアモジュールが20を超える、明らかに複数サブシステムがある、ディレクトリ階層が複雑、入口が分散していて一度で信頼できる全体理解ができない場合は、まずコアモジュールをカバーする。
- コアモジュールは、入口参照チェーン、主要業務ディレクトリ、ビルド設定、CIの到達範囲、テストカバレッジ密度、READMEで明示されたモジュールから総合判断する。
- 未カバーのモジュールは `MEMORY.md` の要確認事項または後続提案に書く。

デグレード戦略：

- 規約が不一致の場合、強制的に統一せず現状を記録する。
- アーキテクチャ判断を確認できない場合、`<!-- 要確認 -->` と標記する。
- 履歴文書がソース、ビルド、CI、テストと衝突する場合は後者を優先し、衝突元を説明する。

## ステップ2：分析レポートを出力し確認を待つ

スキャン完了後、必ず先にレポートを出力して停止する。ファイル作成や保守には進まない。レポートには必ず以下を含める：

1. プロジェクト認識サマリー：位置づけ、技術スタック、コアモジュール、現在フェーズ、コア規約。
2. 根拠ソース：各重要判断に少なくとも1つの追跡可能な根拠を列挙する。例：ディレクトリ、設定、入口ファイル、テスト、既存文書。
3. 初期化状態の初期判断：新規 / 半初期化済み / 初期化済み、および判断根拠。
4. 作成または保守予定のファイル一覧：ルートファイル、`.ai/` 母本文書、Copilot / Cursor / Trae 薄アダプタ層を含める。
5. 各ファイルの想定内容サマリー：用途と主内容を一文で説明する。
6. 検出した不確実点、衝突点、要確認事項。
7. 大規模プロジェクトの場合、本ラウンドのカバー範囲と後続backlog。
8. レポート通りに初期化または保守を続行してよいか、ユーザーに明確に確認する。

レポート末尾には必ず以下を書く：

```text
上記レポートに従って初期化または保守を続行してよいか確認してください。確認前にファイルの作成や変更は行いません。
```

ユーザー確認前は、後続ステップの実行を禁止する。

## ステップ3：初期化状態を判断する

以下のファイルを確認する：

```text
AGENTS.md
MEMORY.md
AI_BOARD.md
.ai/instructions/ai-workflow.md
.ai/docs/README.md
.ai/skills/README.md
.ai/agents/README.md
.ai/prompts/README.md
.github/copilot-instructions.md
.cursor/rules/
```

状態に応じて実行：

- 新規プロジェクト：コアファイルの大半が存在しない。先に `AGENTS.md`、`MEMORY.md`、`AI_BOARD.md` を作成し、次に `.ai/`、最後にエディタアダプタを作成する。
- 半初期化済みプロジェクト：不足部分のみ補完し、既存内容はマージして保守する。
- 初期化済みプロジェクト：古い、衝突している、不足している、または現在のソースと一致しない内容のみ更新する。

## ステップ4：ルートファイルを作成または保守する

### `AGENTS.md`

`AGENTS.md` は唯一のプロジェクト案内と自動ロード入口であり、300行以内に収める。タイトルは以下を使う：

```md
# Project Guidelines
```

必ず含める：

- Project Summary：プロジェクトの位置づけを1〜2文で説明。
- Technology Stack：言語、フレームワーク、ビルドツール、実行依存。
- Core Modules：主要モジュールのみを表で記載。
- Core Conventions：4〜8個のプロジェクト硬性規約。
- Required Reading：少なくとも `MEMORY.md`、`AI_BOARD.md`、主要 `.ai/instructions/*.md`。
- On-Demand Resources：`.ai/docs/`、`.ai/skills/`、`.ai/agents/`、`.ai/prompts/` の主要ファイルとトリガー条件。
- AI Identity Model：`frontend-ai`、`backend-ai`、`test-ai`、`review-ai` を定義。
- Mandatory Workflow：編集前にメモリとボードを読む、操作前にアイデンティティを宣言する、実質的変更後にメモリを更新する、協業やブロック時はボードを更新する。
- Test Gate：実際の技術スタックに応じて test、lint、build、smoke、E2E、lab検証要件を列挙。
- Language：コード、ログ、設定、プロトコルフィールド、ユーザー向け文書の言語要件。

`AGENTS.md` はサマリーとナビゲーションのみを担い、詳細規約本文をコピーしない。

### `MEMORY.md`

`MEMORY.md` は現在のプロジェクトメモリであり、完全な履歴百科ではない。必ず含める：

```md
# MEMORY

Last updated: YYYY-MM-DD HH:MM <timezone>

## 現在フェーズ

## 完了事項

## AIエンジニアリング状態チェックリスト

## 後続提案

## 要確認

## 変更履歴
```

要件：

- 現在フェーズは実際のプロジェクト状態に基づく。
- 100行以内を推奨する。長くなりすぎた場合は `.ai/docs/changelog.md` または `.ai/docs/archive/` に古い履歴をアーカイブする。
- 変更履歴は1行形式を使う：

```text
- YYYY-MM-DD HH:MM | <model-name> | <本変更を一文で説明>
```

- ローカルタイムゾーンを使い、分単位まで記録する。
- 説明は80字以内。
- 不確実な内容は `<!-- 要確認 -->` と標記する。

### `AI_BOARD.md`

`AI_BOARD.md` は唯一のアクティブ協業ボードである。必ず含める：

```md
# AI Collaboration Board

## ファイルモデル

## AI Identities

## Active Board

## Board Rules

## Activity Log Format

## Activity Log
```

Active Boardの表形式：

```md
| ID | Owner | Type | Status | Item | Next Action |
| --- | --- | --- | --- | --- | --- |
```

有効なアイデンティティ：

| Identity | Owner | Scope | Boundary |
| --- | --- | --- | --- |
| Frontend AI | `frontend-ai` | フロントエンドUI、Wails/IPC呼び出しラッパー、三言語文言、ビジュアルとインタラクション。 | 同期コアを実装しない。RabbitMQ/MySQL/HTTPへ直接アクセスしない。 |
| Backend AI | `backend-ai` | バックエンド、サービス、同期コア、設定、DTO、インストーラ、MCP。 | フロントエンド可視のAPI/DTO/エラー意味論の変更は必ずボードに書く。 |
| Test AI | `test-ai` | ユニットテスト、smoke、E2E、lab、負荷テスト、リリースゲート証拠。 | デフォルトではプロダクト挙動を変更しない。テストfixture修正は範囲を説明する。 |
| Review AI | `review-ai` | アーキテクチャレビュー、リリース評価、リスク一覧、横断的な微調整。 | 先にリスクとギャップを列挙する。アイデンティティ横断の変更は理由と影響を説明する。 |

Board Rulesには必ず含める：

1. 作業開始前にアイデンティティを宣言し、その後Active Boardを読む。
2. 新しい問題はActive Boardに追加し、有効な `Owner` に割り当てる。
3. 解決後は `closed` に変更し、Activity Logを追記する。
4. 進められない場合は `blocked` に変更し、何が不足しているかを書く。
5. API、DTO、エラー意味論、ページ範囲の変更は、先にActive Boardを更新し、その後安定契約文書へ同期する。
6. `test-ai` は検証コマンド、成功/失敗結果、未実行理由を記録する。
7. `review-ai` はリスク、ギャップ、ブロッカーを優先して列挙する。
8. 最終返信ではアイデンティティ、処理したBoard項目、残るopen/blocked項目を説明する。
9. 完了履歴が長くなりすぎた場合、`review-ai` が `.ai/docs/archive/` またはフェーズまとめへアーカイブする。

Activity Log形式：

```text
- YYYY-MM-DD HH:mm | <frontend-ai/backend-ai/test-ai/review-ai> | <question/decision/answer/blocker/review/test> | <影響範囲> | <open/closed/blocked>
```

## ステップ5：`.ai/` 母本文書体系を作成または保守する

必須ディレクトリ：

```text
.ai/
  instructions/
  docs/
    archive/
  skills/
  agents/
  prompts/
```

各ディレクトリには必ず `README.md` を置き、以下を説明する：

- ディレクトリ用途
- ファイル命名規約
- いつ新規ファイルを作るか
- いつ新規ファイルを作らないか

必須ファイル：

```text
.ai/instructions/ai-workflow.md
.ai/docs/README.md
.ai/skills/README.md
.ai/agents/README.md
.ai/prompts/README.md
```

必要に応じて作成：

```text
.ai/instructions/<domain>.md
.ai/docs/<topic>.md
.ai/skills/<skill-name>/SKILL.md
.ai/agents/<name>.agent.md
.ai/prompts/<name>.prompt.md
```

必要に応じて作るファイルは、プロジェクト上の根拠が必要である。構造を埋めるためだけの空洞ファイルを作らない。

### `.ai/instructions/*.md`

frontmatterが必須：

```yaml
---
description: "Use when: トリガーキーワード説明"
applyTo: "glob パターン"
---
```

要件：

- 1ファイル1関心事とする。例：バックエンド、フロントエンド、API、テスト、同期アーキテクチャ。
- 内容は短く実行可能にする。
- `AGENTS.md` のプロジェクトサマリーをコピーしない。
- `MEMORY.md` と `AI_BOARD.md` との関係を必ず説明する。
- フロントエンド、バックエンド、テスト、レビュー協業に関係する場合、ルート `AI_BOARD.md` の読取を必須にする。

### `.ai/docs/*.md`

用途：

- 安定した設計文書
- アーキテクチャ説明
- API契約
- フェーズまとめ
- 完了記録
- アーカイブ資料

禁止：

- `.ai/docs/` にアクティブボードを作らない。
- `.ai/docs/` でopen/blocked作業項目を管理しない。
- 失効した内容は `.ai/docs/archive/` へ移す。

### `.ai/skills/<skill-name>/SKILL.md`

frontmatterが必須：

```yaml
---
name: skill-name
description: "Use when: キーワード豊富なトリガー説明"
argument-hint: "入力ヒント"
---
```

以下をすべて満たす場合のみskillを作成する：

- プロジェクトで3回以上出現する反復操作がある。
- 入力と出力が明確。
- 手順が安定して再現可能。
- プロジェクト固有の制約を含む。
- 将来の類似タスクに再利用できる。

「コードを読む」「バグを修正する」のような汎用で空洞なskillは作成しない。

### `.ai/agents/<name>.agent.md`

frontmatterが必須：

```yaml
---
description: "Use when: トリガー説明"
name: "agent-name"
tools: [read, search]
---
```

要件：

- 少なくとも1つの読み取り専用レビューAgentを作成する。
- 役割、責務、境界、やらないことを明確にする。
- toolsは最小必要セットを使う。
- レビュー系Agentはデフォルトで読み取り専用。
- アイデンティティモデルに関係するAgentを作る場合、`frontend-ai`、`backend-ai`、`test-ai`、`review-ai` のいずれかに対応させる。

### `.ai/prompts/<name>.prompt.md`

frontmatterが必須：

```yaml
---
description: "用途を一文で説明"
agent: "agent"
tools: [read, edit, search]
---
```

要件：

- Markdownリンクで関連instructions / skills / agentsを参照する。
- プロジェクトで最も一般的な3〜5個の操作に対して作成する。
- 数合わせのために低価値promptを作らない。
- プロジェクトのシナリオが少ない場合は3個未満でもよい。
- アイデンティティ入口promptでは、先にアイデンティティを宣言し `AI_BOARD.md` を読むことを要求する。

## ステップ6：深度レビューフロー

深度レビューフローはデフォルトでは作成しない。ユーザーが「レビューフローを有効化」「深度レビューを追加」「/review」「深度レビューAgentを作成」などのトリガーを明示した場合のみ、増分で作成する：

```text
.ai/agents/deep-review.agent.md
.ai/prompts/deep-review.prompt.md
tests/README.md または既存テストディレクトリ内の ai-review 説明
MEMORY.md 内の ## レビューログ セクション
```

深度レビューAgentの最小toolsは `[read, search, edit, run]` とし、作業ループは固定：

1. レビュー：`.ai/instructions/`、`AI_BOARD.md`、最近のdiffに基づき問題一覧を出力する。
2. テスト追加：未カバーの重要パスにテストを追加する。空テストは書かない。
3. テスト実行：プロジェクトのテストコマンドを実行し、失敗を記録する。
4. 失敗修正：失敗を最小限に修正する。テスト削除やskipで通過を偽装しない。
5. 再レビュー：レビューへ戻り、連続2ラウンドで新規問題なし、かつ全テスト成功となるまで続ける。

各ラウンド終了時に `MEMORY.md` の `## レビューログ` へ追記する：

```text
- YYYY-MM-DD HH:MM | <model-name> | round <N> | <本ラウンドの変更と結論を一文で説明：合格 / 次ラウンドへ継続>
```

## ステップ7：エディタアダプタ層

`.ai/` は母本であり、エディタアダプタ層は薄い入口のみとする。完全な規約をコピーしない。

### Copilot

作成または保守：

```text
.github/copilot-instructions.md
.github/instructions/*.instructions.md
```

`.github/copilot-instructions.md`：

- 10〜20行。
- コア入口のみを書く。
- 必ず先に `AGENTS.md` を読むよう要求する。
- `MEMORY.md` と `AI_BOARD.md` に必ず言及する。
- 完全な規約をコピーしない。

`.github/instructions/*.instructions.md`：

- コア `.ai/instructions/*.md` に対して2〜4個のアダプタのみ作成する。
- frontmatterが必須：

```yaml
---
applyTo: "glob パターン"
---
```

- 本文には、どの母本ファイルを読むか、および2〜4個の硬性ルールのみを書く。

### Cursor

作成または保守：

```text
.cursor/rules/00-core.mdc
.cursor/rules/<domain>.mdc
```

`00-core.mdc`：

- `alwaysApply: true`
- `globs: ["**/*"]`
- 入口のみを書き、完全な規約をコピーしない。

分野別ルール：

- `alwaysApply: false`
- `globs` は対応する `.ai/instructions/*.md` の `applyTo` と一致させる。
- 母本リンクと少数の硬性ルールのみを書く。

### Codex

Codexはルート `AGENTS.md` をネイティブに読むため、追加アダプタファイルは不要。

### Trae

Traeはデフォルトで `AGENTS.md` と `.ai/` 母本を再利用する。プロジェクトに既存のTraeネイティブルールディレクトリがある場合、またはユーザーが明示的に要求した場合のみ、Traeの薄い入口を作成する。

Traeアダプタ規則：

- 既存のTraeネイティブルールディレクトリがある場合、その構造を保持し、母本リンクのみ補足する。
- Traeルールパスを確認できない場合、`<!-- 要確認 -->` と標記し、ディレクトリを勝手に作らない。
- Traeアダプタ層は完全な規約をコピーしてはならない。
- Traeアダプタ層は第二のアクティブボードを作ってはならない。

## ステップ8：旧ボード移行

旧アクティブボードが以下にある場合：

```text
.ai/docs/ai-collaboration-log.md
```

次のように処理する：

- まだ有効なopen/blocked項目をルート `AI_BOARD.md` へ移行する。
- 旧ファイルには移行告知のみを残す。
- 旧ファイルに新しいopen/blocked項目を追加しないことを明記する。
- すべての案内をルート `AI_BOARD.md` へ向ける。

旧ファイル内容例：

```md
# AI Collaboration Board moved

本ファイルはアクティブボードではありません。アクティブなAI協業ボードはリポジトリルートへ移動しました：

AI_BOARD.md

ルール：
- `AI_BOARD.md` が唯一のアクティブボードです。
- `.ai/docs/` は安定文書、完了記録、アーカイブ資料のみを保持します。
- 本ファイルにopen/blocked項目を追加しないでください。
```

## ステップ9：安全書き込み規則

- 既存ファイルはマージを優先し、盲目的に上書きしない。
- 元内容が現在の分析と衝突する場合、元内容を保持し `<!-- 要確認 -->` と標記する。
- 不確実な内容は必ず `<!-- 要確認 -->` と明示する。
- 同種の文書が既にある場合、新規文書は既存文書へリンクし、大量に重複記述しない。
- 空の業務文書を作らない。
- ビルド成果物、キャッシュ、生成コードを現在規約の根拠にしない。
- パス変更時はすべての参照を同期する。
- アダプタ層は常に母本より短くする。
- `AI_BOARD.md` と競合するアクティブボードを作らない。

## ステップ10：完了後セルフチェック

形式チェック：

- `AGENTS.md` が存在し、300行以内。
- `MEMORY.md` が存在し、最終更新時刻がある。
- `AI_BOARD.md` が存在し、唯一のアクティブボードである。
- `.ai/` ディレクトリ構造が完全。
- 各READMEが用途と作成条件を説明している。
- instructions / skills / agents / prompts に正しいfrontmatterがある。
- skillの `name` がディレクトリ名と一致する。
- descriptionが `Use when: ...` を使っている。
- `.ai/docs/` にアクティブボードがない。
- `.ai/docs/ai-collaboration-log.md` が存在する場合、移行告知のみである。
- Copilot / Cursor / Traeアダプタ層が入口のみである。
- 存在しないファイルへの参照がない。

内容信頼性チェック：

- 各instructionにプロジェクト上の根拠がある。
- 各skillが実在する高頻度フローに対応している。
- 各promptが実在するinstruction、skill、agentのいずれかを参照している。
- モジュール分割が実際のディレクトリ構造と入口ファイルに一致している。
- 未確認内容がすべて `<!-- 要確認 -->` と標記されている。
- ビルド成果物、生成コード、履歴残存物を現在規約の根拠にしていない。

協業チェック：

- Active BoardのOwnerが有効なアイデンティティのみを使っている。
- Active Boardのopen/blocked項目が明確。
- Activity Log形式が統一されている。
- 最終返信でアイデンティティ、処理したBoard項目、残るopen/blocked項目を説明している。
- 実質的変更が `MEMORY.md` に記録されている。
- 協業、ブロック、API、DTO、エラー意味論、ページ範囲変更時に `AI_BOARD.md` を更新している。

深度レビューチェック：

- 深度レビュー未トリガー時は、`deep-review.agent.md`、`deep-review.prompt.md`、レビューテストディレクトリを作成していない。
- 深度レビューがトリガーされた場合、レビューAgent、Prompt、テスト説明、`MEMORY.md ## レビューログ` が同期済み。

## 完了出力

最後に出力する：

1. 判断したプロジェクト状態：新規 / 半初期化済み / 初期化済み。
2. 作成または更新したファイル一覧。
3. 現在の `AI_BOARD.md` のopen/blocked項目。
4. セルフチェック結果サマリー：合格項目と未合格項目。
5. 要確認事項まとめ。
6. 後続提案。
